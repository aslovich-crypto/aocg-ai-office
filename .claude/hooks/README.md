# Защита Claude Code от опасных операций

Два слоя, чтобы Claude Code не мог по ошибке тронуть секреты, прод-БД
или запушить без твоего ведома.

## Слой 1 — deny-список (`.claude/settings.json`)

Блокирует по шаблону: чтение секретов (.env, .ssh, ключи), деструктив
(rm -rf, git reset --hard, git push, sudo), инлайн-DDL на проде.
Откат несохранённых правок (git checkout/restore) — спрашивает каждый раз.

## Слой 2 — хук (`block-prod-db.sh`, PreToolUse на Bash)

Закрывает то, что шаблоны не видят: сканирует СОДЕРЖИМОЕ запускаемых
.py/.sql и блокирует деструктивный SQL ТОЛЬКО ЕСЛИ скрипт подключён
к боевой базе (railway / asyncpg / DATABASE_URL / up.railway.app).
Локальные скрипты не трогает. Чистая инспекция (git show/diff/log/blame,
grep, cat, head, tail, less) не блокируется, даже если читаемый файл полон
SQL — фаерит только при наличии исполнителя (python / psql / dropdb / пайп
в sh|bash), включая `… | python`.

## Когда временно снимать защиту

Push и миграции схемы БД заблокированы намеренно — это осознанные операции.

**Деплой:** push делаешь вручную. Либо на время закомментируй правило
`git push` в settings.json и верни после.

**Миграция БД** (DROP/CREATE индекса и т.п.):

1. В settings.json сними правило DDL + отключи хук (или через `/hooks`).
2. Прогони BEGIN/ROLLBACK-валидацию, затем применение.
   ⚠️ Хук блокирует и саму валидацию — он не отличает ROLLBACK от COMMIT
   по тексту. Поэтому защиту снимаешь на весь цикл «валидация → применение».
3. Сразу верни защиту обратно.

## Управление

- Посмотреть/переключить правила и хук: меню `/hooks` или правка
  `.claude/settings.json`.
- Если хук не подхватился — открой `/hooks` один раз (перечитает конфиг).
