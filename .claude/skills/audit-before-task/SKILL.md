# Аудит перед большой задачей

Перед изменением схемы БД, интеграцией или крупной правкой — сначала аудит текущего состояния. Только потом рабочие правки. Выбрать шаблон по типу задачи.

## Когда применять
- Изменение схемы БД, миграции
- Интеграция внешнего API (банк, ФНС, OAuth, хранилище)
- Крупная UI-правка, изменение форм
- Любая задача, где важно знать точное текущее состояние

## Шаблон А — перед изменением СХЕМЫ БД
Собрать и показать, ничего не меняя:
- Точные схемы затрагиваемых таблиц через `information_schema.columns`
- Существующие индексы через `pg_indexes`
- Эндпоинты, которые читают/пишут эти таблицы
- Формат POST/GET для этих эндпоинтов
- Существующие тесты на эти пути
- Прод-маркеры (как проверить, что изменение доехало)
- HEAD-коммиты обоих репозиториев

## Шаблон Б — перед изменением UI/UX
Собрать и показать:
- Затрагиваемые компоненты и их расположение
- State-машины (phase, useState) на этих экранах
- Обработчики событий
- Тексты, которые видит пользователь
- Условия рендеринга
- Поведение при ошибках

## Шаблон В — перед интеграцией
Собрать и показать:
- Существующие env-переменные в реестре
- Документацию партнёра (эндпоинты, формат токенов, sandbox)
- Используемый SDK / способ вызова
- Целевые эндпоинты интеграции
- Точки соединения с фронтендом

## Правило
Аудит — это режим «только чтение». НЕ менять код на этом шаге. Дождаться явного «ок» перед переходом к правкам.
