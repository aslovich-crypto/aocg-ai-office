# Чеклист безопасности перед деплоем

Перед `git push` в main для критичных задач (auth, БД, файлы, API-ключи) — пройти чеклист. pre-commit хуки ловят секреты автоматически, этот чеклист — про логику, которую машина не проверит.

## Чеклист
- [ ] Нет секретов в diff? (дублирует pre-commit, но перепроверить)
- [ ] Все новые пользовательские входы валидируются?
- [ ] Все SQL-запросы параметризованы (`$1`, `$2`), без f-strings и конкатенации?
- [ ] Нет логирования чувствительных данных (пароли, ИНН, fn, карты)? Маскирование на месте?
- [ ] CORS не открыт для `*`, только whitelist?
- [ ] Новые эндпоинты требуют авторизации, где нужно?
- [ ] org-scope фильтрация во всех запросах к данным (нет доступа к чужим объектам по id)?
- [ ] Rate limiting на критичных эндпоинтах (auth, fns, receipts)?
- [ ] Ошибки не выдают информацию об инфраструктуре (traceback скрыт в проде)?
- [ ] Логин не различает «нет пользователя» и «неверный пароль» (защита от энумерации)?
- [ ] Обновлены тесты безопасности (auth, rate limit)?
- [ ] Документация обновлена?

## Особое внимание
- Чувствительные поля для маскирования: пароли, password_hash, ИНН, fn, номера карт, cvv, паспорта.
- При работе с персональными данными — обсудить безопасность ДО написания кода.
- При новой зависимости — проверить размер, зависимости и известные CVE.

## Правило
При работе с чувствительными данными — остановиться и обсудить безопасность до кода, а не после. Если замечена нереализованная лучшая практика — предложить её, не молчать.
