# Валидация миграции БД через BEGIN/ROLLBACK

Перед любой схема-изменяющей миграцией на проде — прогнать новый DDL внутри транзакции, которая в конце откатывается. Ничего не сохраняется, но синтаксис и идемпотентность проверены реальным PostgreSQL.

## Зачем
`init_db()` выполняется в FastAPI lifespan на каждом старте контейнера Railway. Синтаксическая ошибка в DDL = приложение не поднимется = прод лежит. Валидация ловит это заранее без риска.

## Как (через asyncpg, не psql)
```python
tr = conn.transaction()
await tr.start()
await conn.execute(NEW_DDL)          # 1. проверка синтаксиса
await conn.execute(NEW_DDL)          # 2. идемпотентность (вторая прогонка = no-op)
# 3. проверить результат через information_schema.columns / pg_indexes
await tr.rollback()                  # 4. ОТКАТ — никогда не COMMIT
# 5. контрольный SELECT после отката: схема не изменилась
```

## Правила
- НИКОГДА не COMMIT в валидационном скрипте — только rollback.
- DDL прогоняется дважды: первый раз — синтаксис, второй — идемпотентность.
- После отката — контрольная проверка, что схема осталась прежней.
- Запускать через Railway-окружение с доступом к проду (psql локально не используется).
- Перед миграцией также показать обратный DDL (точку отката): DROP COLUMN / DROP INDEX / DROP TABLE.

## Применимость
Все миграции схемы: новые колонки, таблицы, индексы, переименования. Также — валидация новых SELECT-запросов перед выкаткой.
