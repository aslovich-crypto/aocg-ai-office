# AOCG AI Офис — бэкенд · инструкции для AI-агента

Постоянные правила для AI-агента (Claude Code). Соблюдать во всех сессиях. Это «оперативная память» репозитория — обновляется вместе с кодом.

## Что это за проект
Бэкенд платформы AOCG AI Офис — B2B SaaS для управления первичными финансовыми документами (чеки, авансовые отчёты, акты/УПД, счета) и управленческим учётом для российского малого и среднего бизнеса. Оператор персональных данных — ИП Шукалович Алексей Иванович.

## Технологический стек
- Язык: Python
- Фреймворк: FastAPI
- База данных: PostgreSQL через asyncpg (пул соединений)
- Хостинг: Railway (auto-deploy из main)
- CI: GitHub Actions, pytest на каждый push

## Команды
- Запуск локально: `uvicorn app.main:app --reload`
- Тесты: `pytest tests/ -v`
- Виртуальное окружение: `source venv/bin/activate`

## Ключевые файлы
- `app/main.py` — FastAPI, CORS (whitelist), startup, миграции
- `app/database.py` — asyncpg pool, init_db()
- `app/auth.py` — JWT, bcrypt, блокировка после попыток, get_current_user
- `app/routers/` — эндпоинты: receipts, reports, cards, fns, ocr, users, invite, auth
- `app/categorization.py` — авто-категоризация
- `tests/test_api.py` — тесты эндпоинтов
- `tests/conftest.py` — FakePool, зеркалит поведение PostgreSQL

## Структура репозитория и лимиты на размер
Куда класть новый код (раскладка обязательна, не импровизировать):
- Эндпоинты — в `app/routers/`, один файл = один домен (receipts, reports, categories...). Новый домен — новый файл роутера, не дописывать в чужой.
- Бизнес-логика (категоризация, парсеры, расчёты) — отдельными модулями (`app/categorization.py`, `app/parsers/`), НЕ внутри роутеров. Роутер только принимает запрос, зовёт логику, возвращает ответ.
- `app/main.py` — только сборка приложения (CORS, startup, подключение роутеров). Не складывать сюда бизнес-логику.
- Общие утилиты — `app/utils/` или отдельный модуль, не дублировать функцию между файлами.

Лимиты (это сигналы пересмотреть, не жёсткие запреты):
- Файл роутера > ~300 строк → сигнал, что в нём смешаны домены или логика просочилась из сервисного слоя. Предложить разделение.
- Функция/хендлер > ~50 строк → сигнал вынести часть в отдельную функцию.
- Дублирование одной логики в 2+ местах → вынести в общую функцию.
- При приближении к лимиту — не молчать, а сообщить и предложить, как разнести.

## Терминология (строго)
- Платформа — AOCG AI Офис целиком
- Приложение — Прима, Финансы
- Модуль — только внутри Финансов (ДДС, ОПУ, Бюджет)
- Раздел — внутри Примы (Чеки, Авансовые отчёты, Акты, Счета)
- Слово «модуль» не применять к Приме

## Конвенции кода
- Только параметризованные SQL-запросы asyncpg (`$1`, `$2`). НИКОГДА не строить SQL через f-strings или конкатенацию.
- Поле `source` в receipts: `fns` | `qr_scan` | `photo_ocr` | `manual`.
- Defensive programming: в if/elif по HTTP-кодам — всегда catch-all `else`.

## Паттерны «делай»
- Поиск «где/как у нас сделано X» и обзор по многим файлам — отдавать субагенту Explore (параллельно, если направлений несколько), а не читать файлы в основном потоке. Быстрее и не раздувает контекст сессии.
- org-scope фильтрация во ВСЕХ запросах к данным (защита от доступа к чужим объектам по id). Каждый эндпоинт проверять на cross-org доступ.
- Параметризованные запросы везде.
- При смене продакшен-логики INSERT/SELECT — зеркально править FakePool в conftest.py.
- Контрактный тест keys_match между параллельными парсерами одного INSERT-пути (fns_parser, ocr_parser и будущие): множества ключей dict должны совпадать.
- Атомарные коммиты, Conventional Commits.

## Паттерны «не делай»
- НЕ коммитить секреты — только в Railway → Variables. `.env` в `.gitignore`.
- НЕ логировать в открытом виде: пароли, ИНН, fn (фискальные номера), полные номера карт, паспорта. Маскировать: карты `****1234`, fn — только факт наличия.
- НЕ использовать `*` в CORS на проде — только whitelist в `CORS_ORIGINS`.
- НЕ возвращать traceback в ответах API при `ENVIRONMENT=production`.
- НЕ выдавать «пользователь не найден» при логине — единое «Неверный логин или пароль» (защита от энумерации).
- НЕ делать батч-рекатегоризацию без фильтра `WHERE category_manual = FALSE` (защита ручных правок пользователя).

## Безопасность
- Секреты: только Railway Variables, минимальные scope токенов.
- bcrypt (стоимость 12+) для паролей.
- JWT: access 60 мин, refresh 30 дней, HS256; при logout — в revoked_tokens.
- Блокировка: 5 неверных попыток → 15 минут (failed_attempts, locked_until).
- Rate limiting на auth и критичных эндпоинтах.
- 152-ФЗ: согласие при регистрации — два НЕзаполненных чекбокса, лог в user_consents (user_id, время, IP, версия политики, текст).

## Хранение данных и резидентность (152-ФЗ)
- ПД граждан РФ хранятся физически в РФ: БД (PostgreSQL) и фото чеков
  (объектное хранилище). Целевой хостинг — Yandex Cloud (задача S-06).
- Объектное хранилище для фото и файлов — Yandex Object Storage
  (S3-совместимое). Cloudflare R2 и любые зарубежные хранилища ПД —
  ЗАПРЕЩЕНЫ (противоречат 152-ФЗ).
- Трансграничная передача ПД (OCR через Anthropic Claude Vision (США);
  внешняя почта) допустима ТОЛЬКО при согласии субъекта на трансграничную
  передачу + уведомлении Роскомнадзора. Это обработка, не хранение —
  за рубежом данные не оседают.
- НЕ добавлять новые внешние сервисы, получающие ПД, без отметки в
  юридическом контуре (Согласие на обработку ПД + уведомление РКН).
  При выборе любого внешнего сервиса для ПД — проверять страну хранения.

## Рабочий процесс
- Одна фича — одна сессия, один атомарный коммит.
- Сначала план (plan mode), код не писать до одобрения.
- Показывать diff до применения; коммит — отдельной командой после ручной проверки.
- Перед любой миграцией схемы БД — BEGIN/ROLLBACK валидация через asyncpg (см. skill db-migration-validation). НИКОГДА не COMMIT в валидационном скрипте.
- Перед миграцией — показать обратный DDL (точку отката: DROP COLUMN / DROP INDEX / DROP TABLE).
- Перед большой задачей — аудит текущего состояния (см. skill audit-before-task).

## Формат коммитов
`тип(scope): краткое описание`
Типы: feat, fix, chore, docs, refactor, test.
Scope: schema, dedup, privacy, scan, parser, auth и т.д.
Пример: `feat(schema): add receipt_items table (task #7)`

## Чего НЕ касаться без явного запроса
- Конфигурацию деплоя Railway и переменные окружения.
- Схему БД в продакшене (только через валидированную миграцию).
- Сторонние интеграции (ФНС-прокси, Альфа-Банк) без обсуждения формата.
